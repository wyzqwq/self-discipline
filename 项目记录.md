## 一、项目是什么

把 edmond《如何提升自制力》里的两代自控方法论做成一个能真正用的程序（PWA）。

**源材料**：`如何提高自制力.pdf`（49 页，2025-07-22）。核心是两套技术：

- **CTDP（链式时延协议）**——第一代，管"当下要不要做"。
  - 神圣座位原理：链式 #N 计数、失败清零、非线性价值压缩（连续做到的价值越往后越贵，不舍得断）。
  - 下必为例原理：判例法思维，"运动员"（执行）与"裁判"（定规则）分离。
  - 线性时延原理：15 分钟预约辅助链，破解启动困难（先约定 15 分钟后开始，降低启动门槛）。
- **RSIP（递归稳态迭代协议）**——第二代，管"长期结构性自控"。
  - 定式树 / 国策树：每天最多加一个定式。
  - 堆栈式级联删除：一条定式失败，连带删掉它的所有子节点（子节点是建立在父定式成立的前提上的）。
  - 定式 = 针对某个反复出现的负面状态的"局部最优解"。
  - "胜于易胜"：从容易赢的地方开始，稳态跃迁 E0→E1→E2。

**成品**：`RSIP.html`，单文件 PWA（约 1354 行 / 159KB）。也同步为 Cowork artifact，id = `rsip-ctdp-console`。

## 三、数据模型

`localStorage` 键 `rsip_ctdp_state_v1`，结构：

```
state = {
  tree: {
    nodes: [ ...定式节点 ],
    lastAddedDate: "YYYY-MM-DD" | null,   // 最近一次"加子定式"占用的逻辑日
    migratedDayBoundary: true,            // v1.2 一次性迁移标记
  },
  ctdp: { main:{count}, aux:{count}, activeAux, activeFocus, ... },  // 链式计时
  _ts: 时间戳,                             // 云同步 last-write-wins 用
}
```

**定式节点字段**：
- `id` / `idx`（序号，诊断挂载编号引用它）
- `content`：规则内容（"进门 15 分钟内必须洗澡"）
- `name`：定式名，可选（"洗澡令"）
- `parentId`：父节点 id；**为空 = 根定式**
- `checkMode`：`'sameday'`（当天确认，默认）| `'nextday'`（次日回顾昨天）
- `createdAt`：ISO 时间戳
- `status` / `log`：`log` 是 `{ "YYYY-MM-DD": "success"|"fail" }`
- 已废弃字段：`problem` / `intervention`（migrate 会补 `name=''`，不再展示）

## 四、核心机制（重要，容易踩坑）

### 1. 单根模型
全树只允许**一个**根定式（`parentId` 为空），代表对整棵定式树的治理/管理要求。
- `canAddRoot() = !hasRoot()`：加根**不占**"每日一个"名额、**不改** `lastAddedDate`、加完就不能再加第二个根。
- 子定式（挂在别人下面）受每日一个限制：`canAddChildToday() = lastAddedDate !== 今天`。
- `canAddToday() = canAddChildToday() || canAddRoot()`。

### 2. 逻辑日边界：早上 6:00 翻天（关键）
沿用 `HARNESS_FOR_ALL.md` 的日本 30 小时制——**一天从早上 6:00 开始**，0:00–5:59 仍算前一天（深夜熬夜不跳天）。**不用北京时间 00:00。**
- `DAY_START_HOUR = 6`；`dayStrOf(date)` = 把时间减 6 小时再取本地年月日。
- `todayStr / dateStrOffset / yesterdayStr / checkReady` 及 CTDP"今日专注"计数全走 `dayStrOf`。
- ⚠️ `const DAY_START_HOUR = 6` 必须定义在 `let state = load()` 之前，否则 TDZ 报错（load 会调 migrate → dayStrOf）。

### 3. 打卡时机 checkMode
- `sameday`（默认）：当天就能确认。打卡目标日 = 今天。
- `nextday`（次日回顾）：像"上床不带手机"这种当天无法判断的，第二天回顾昨晚。打卡目标日 = 昨天。
- `checkDateFor(n)` = nextday 时 `yesterdayStr()`，否则 `todayStr()`。
- `checkReady(n)` = nextday 且 createdAt 是今天 → false（今天刚建的次日节点，明天起才能回顾）。
- **只确认昨天，不补多天**：nextday 每次只盯"昨天"这一格；漏了好几天，中间的补不回来，只能记最近一天。
- 打卡列表是**按定式**渲染的，一条定式永远只有一个打卡入口，不会因为漏打卡而堆积成多条待办。

### 4. 级联删除 + 删除退名额
- 打卡失败（主动点 ✗）→ **立即**删除该节点及其整个子树（`subtreeIds` DFS）。不是"第二天"删，是当场删。
- 漏打卡**不会**自动判失败、不会自动删；定式一直留着等你确认。
- **删除退名额**：手动删除时若被删节点含"今天刚加、正占今日名额"的子定式（parentId 非空且 `dayStrOf(createdAt)===今天===lastAddedDate`），把 `lastAddedDate` 置 null 退还名额（视为撤销/改错）。helper = `refundDailyIfContains`。
  - 只有**手动删除**退；**失败级联删除不退**（失败是惩罚，不因删除免责）；删根不涉及（根不占名额）。

### 5. AI Agent（起名 + 子定式建议 + 诊断）
- 双路：桌面端优先 `window.cowork.askClaude`；否则只要配了 Supabase 就走云函数中转，**手机端也能用**。`hasAgent() = hasCowork() || 有 syncCfg`。
- **手动模式**：挂载 select + 内容框（AI 建议内容，基于所选父节点提子定式建议）+ 名字框（AI 起名）。
  - **AI 起名带祖先链**：起名 prompt 会从当前挂载父节点（`addParent` 的值）回溯到根，拼出"根→…→父"的上级脉络喂给 AI，让名字呼应所属主题、成体系。新增子定式和重命名共用此逻辑（重命名时 `addParent` 已预填为该节点的父，自动生效）。
  - **AI 建议内容带加权祖先脉络**：内容建议 prompt 用 `weightedAncestorText(parentId)` 拼出从近到远的祖先，每层标注归一化权重百分比——越靠近的祖先权重越大，**指数递减**（`ANCESTOR_DECAY=0.5`：父 d=0 权重最大，祖父减半…再归一化）。prompt 要求 AI **按标注的百分比配比参考**：权重高的影响大，但低权重的上层也必须体现、不可忽略（避免把上层当可选的"兼顾项"架空百分比）。此外 prompt 要求生成的子定式建议**与已有兄弟定式明显不同**（禁同义改写）、且 4 条之间彼此不重复。helper：`ancestorsFromNear(id)`（从近到远含自身）+ `weightedAncestorText(parentId)`。
  - **加根定式的 AI 建议**：根定式全树只有一个，是**统领整棵树的总体纲领/治理原则**（规定整棵树如何建立、执行、复盘、迭代），不是某条针对具体负面状态的普通规则。所以根分支 prompt 让 AI 提的是关于复盘/失败处理/迭代节奏/优先级的元规则；已有定式仅作为"理解用户在治理什么主题"的背景，空树时不塞。
  - 注：v1.3.1 已删掉节点卡片上原来的「AI 建议」按钮——它跟"点该节点的 +子定式 → 表单里 AI 建议内容"功能重复、还易出 bug。要 AI 提子定式，走 +子定式面板即可。
- **AI 诊断模式**：描述今日不自律现象 → AI 返回 4 个方案，格式 `内容|挂载编号|理由`（编号引用现有定式序号，0 = 作为根）；`parseDiag` 校验挂载编号，越界/0 且已有根则回退挂到根节点下；渲染成 4 张卡片，四选一后填入内容 + 挂载父。

### 6. 云函数中转（AI 走火山方舟）
- Supabase Edge Function 名 `ai`，前端 POST 到 `{url}/functions/v1/ai`，带 apikey + Bearer(publishable key)，body `{prompt}`，返回 `{text}`。
- 函数内转发到火山方舟：endpoint `https://ark.cn-beijing.volces.com/api/plan/v1/chat/completions`，model `ark-code-latest`，Bearer = `Deno.env.get('ARK_KEY')`。
- ⚠️ **ARK_KEY 只存 Supabase secret，绝不硬编码进前端**。publishable key 放前端是安全的（设计上就是公开的）。

### 7. Supabase 云同步
- `localStorage` 键 `rsip_sync_cfg_v1` 存 `{url, key}`；表 `sync_state(id='main', data jsonb, updated_at bigint)`。
- `syncPull / syncPush / schedulePush(800ms 防抖)`，冲突用 last-write-wins（比 `state._ts` 与 `cloud.updated_at`）。
- 开机 `initSync()` 拉取，`visibilitychange` 重新拉。导入时强制 `state._ts=Date.now()` 防被覆盖。
- 已预置用户的 Supabase 配置（url = mlthrrozzqqumhpqomgn）。

### 8. 修改定式（白名单式复用添加面板 = 重命名 + 提醒时间）
- 每个节点 actions 里只有一个「修改」按钮（`data-edit`），点开复用 addSheet。整合了原先分开的「重命名」和「提醒」两个按钮。
- `openRenameSheet(id)`：预填当前挂载/内容/名字/checkMode/提醒时间，标题改"修改定式"、确认按钮改"保存"，聚焦并选中名字框。
- **白名单锁**：`setRenameLock(true)` 遍历 addSheet 直接子元素，只放开白名单（`.grip`/`.btn-row`/`addTitle`/`addDesc`/`nameField`/`aiNamePanel`/`remindField`），其余全部加 `.locked`（opacity+pointer-events:none+灰度）。**以后表单新增栏目默认就是锁住的**——不在白名单里就锁，防止漏放开。
- `addConfirm` 顶部有修改分支：`renameId` 非空时只改 `n.name` 与 `n.remindAt` 后返回，不碰其它字段、不新增节点、不占名额。
- `openAddSheet` 里复位：`renameId=null` + `setRenameLock(false)` + 还原标题/描述/按钮文案 + `setRemindPicker('')`（新增默认不提醒）。

### 8b. 提醒时间选择器（滚动式，防越界）
- 提醒时间栏 `#remindField` 在 addSheet 内（新增和修改共用），含一个复选框 `#remindOn`（"到点提醒我确认这条定式"）+ 两个 `<select>` 滚动选择器 `#remindHour`（00-23）/`#remindMin`（每 5 分钟，00/05/…/55 共 12 项）。用原生 select 而非 `input[type=time]`：iOS 上就是滚轮、选项固定绝不越界，不会手打出非法值。
- **每日汇总时间**（`#digestField`，在「提醒设置」sheet 里）也是同款：复选框 `#digestOn` + `#digestHour`/`#digestMin` 滚轮。开关关→清空 `dailyDigestAt`；开关开或改时/分→即时存 `'HH:MM'`。
- 通用 helper：`fillTimeSelect / setTimePicker(prefix,val) / getTimePicker(prefix)`（prefix='remind'|'digest'）；`setRemindPicker/getRemindValue` 是单条提醒的兼容别名。关闭复选框 → 选择器加 `.off`（灰禁用）返回 `''`；开启 → 返回 `'HH:MM'`。预填时把旧数据任意分钟对齐到最近 5 分钟倍数（07:23→07:25）。默认时间 21:00。
- **开任何提醒即自动申请授权**：单条 `remindOn`、汇总 `digestOn` 勾选后都会调 `ensureNotifyPermission()`——支持且已 standalone 时自动 `Notification.requestPermission()`→`registerSW()`；不支持/未加主屏幕/被拒时只给一次轻提示（`_notifyHintShown`），时间照存，等条件满足再生效。不再只能在「开启提醒」按钮那里授权。
- 数据仍存节点字段 `remindAt`（`'HH:MM'` 或空）与 `state.notify.dailyDigestAt`。

### 9. 提醒 / 通知（v1.4，仅适配苹果，分两阶段）
- **只适配苹果**：iOS 16.4+、必须「添加到主屏幕」后从主屏幕图标打开（standalone），用户手动授权。Safari 标签页里收不到推送——苹果强制限制。
- **service worker**：独立文件 `sw.js`（不能内联进 HTML），scope=`/`。`netlify.toml` 给 `/sw.js` 加直通 header（`Service-Worker-Allowed=/` + `no-cache`），`/` 仍重写到 RSIP.html（sw.js 路径不同不受影响）。sw.js 处理 `message`（阶段一本地测试通知）、`push`（阶段二预留）、`notificationclick`（聚焦/打开 App）。
- **阶段一（已做）**：顶部铃铛按钮 `#btnNotify` → 提醒设置 sheet `#notifySheet`。`notifySupported()`（查 serviceWorker/Notification/PushManager）+ `isStandalone()`（matchMedia 做了存在性防御）决定 UI：不支持→隐藏开启按钮；非 standalone→显示"加主屏幕"引导并禁用；已就绪→点「开启提醒」走 `Notification.requestPermission()`→`registerSW()`→postMessage 发一条本地测试通知。开机若已授权自动 `registerSW()`。测试见 `test_notify.js`（3 场景 20 项）。
- **阶段二（已做，后端待部署）**：Web Push 真订阅全套已写好，等用户在 Supabase 网页控制台部署。
  - **前端订阅**：`subscribePush()` —— 已授权且配了云端时，`pushManager.subscribe`（applicationServerKey=VAPID 公钥）→ 拿 endpoint/p256dh/auth，连同 `getDeviceId()`（localStorage 存一台设备一个 UUID）upsert 进 `push_subscriptions` 表（Prefer: merge-duplicates）。同时把 `Intl` 时区写进 `state.notify.tz` 随云同步上去，供服务端按用户本地时钟比对。`unsubscribePush()` 反向清理。在 `ensureNotifyPermission`（授权成功）、`notifyEnable`（开启提醒按钮）、开机（已授权时）三处调用。
  - **VAPID 公钥硬编码在前端**（`VAPID_PUBLIC_KEY`，公开安全）；**私钥只进 Supabase secret `VAPID_PRIVATE_KEY`**，绝不进前端（同 ARK_KEY 原则）。
  - **订阅表** `push_subscriptions`：`device_id`(主键)/endpoint/p256dh/auth/updated_at/created_at/`last_sent`(jsonb 去重)。RLS 宽松 anon all（同 sync_state 单用户策略）。建表 SQL 在 `supabase/push_subscriptions.sql`。
  - **Edge Function `remind-cron`**（`supabase/functions/remind-cron/index.ts`）：扫描+发送合二为一，减少部署步骤。pg_cron 每 5 分钟触发一次。逻辑：读 `sync_state(id='main').data`（就是前端 state）→ 按 `state.notify.tz` 算当前时区的分钟数与 6:00 逻辑日（`nowInTz`，复刻前端 dayStrOf）→ 遍历节点，`remindAt` 落在当前 5 分钟窗口 `(now-5, now]` 且该定式**未确认**（`isUnconfirmed`：sameday 看今天/nextday 看昨天，success 或 fail 都算已确认不打扰）就入队；`dailyDigestAt` 命中窗口则统计未确认条数，>0 才推。→ 取所有订阅逐个用 `web-push`+VAPID 私钥发；`last_sent`(jsonb, key=`node:id:日期:HH:MM`/`digest:日期:HH:MM`) 去重，**同一提醒、同一天、同一时间**只推一次（v1.4.1：键含提醒时间，"推过后往后调时间能再响"，调回今天已推过的旧时刻不重响——符合"重新提醒必指向未来"），并只保留当天键防膨胀；发推遇 404/410 视为失效订阅自动删除。
  - **部署步骤**（网页控制台，不用 CLI）见 `supabase/部署步骤-阶段二.md`：①SQL Editor 建表 ②Settings→Edge Functions 加 3 个 secret（VAPID_PRIVATE_KEY/VAPID_PUBLIC_KEY/VAPID_SUBJECT）③Create function `remind-cron` 贴代码 Deploy ④SQL 里 `cron.schedule` 每 5 分钟 http_post 触发（带 service_role Bearer）⑤手机加主屏幕开提醒完成订阅。
  - **测试** `test_cron.js`（20 项，node 直跑）：时区/6:00 逻辑日翻天（凌晨 05:30 回退前一天、06:00 翻当天）、5 分钟命中窗口边界（08:25 落窗外不重复）、未确认判定（sameday/nextday、success/fail 都算已确认）、汇总计数、去重键 prune。纯逻辑从 index.ts 原样搬出保持同步。
  - 提醒内容：单条定式「该确认「名字」了」、每日汇总「你还有 N 条定式待确认」（N = 需当天确认的今天 + 需次日回顾的昨天里未打卡的）。CTDP 15 分钟预约到点提醒留到修 CTDP bug 时一起做。
- **阶段二步骤 1（已做，前端数据+UI）**：数据模型加两处——节点字段 `remindAt`（`'HH:MM'`，空=不提醒，每条定式各自设）、`state.notify.dailyDigestAt`（每日未确认汇总的推送时间，空=不开）。migrate 补默认。UI：节点操作区**「修改」按钮**（整合了重命名+提醒，白名单式复用 addSheet，见第 8 节）——修改面板里可改名字 + 用**滚动选择器**（时/分 select，防越界，见第 8b 节）设提醒；**新增定式时也能在同一面板一次设好提醒**（默认关）。meta 行显示 `⏰ HH:MM`。每日汇总时间在「提醒设置」sheet 的 `#digestTime`。**「每日未确认」= 需当天确认的今天 + 需次日回顾的昨天**。测试见 `test_remind.js`（31 项）。后端订阅/定时扫仍在阶段二后续步骤。

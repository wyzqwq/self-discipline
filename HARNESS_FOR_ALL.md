# HARNESS_FOR_ALL — 通用协作规范与项目交接清单

> 这是从当前项目协作经验中提炼出的通用版 HARNESS，目标是复用到所有后续项目。每个新项目开始前，应先读取本文件；若项目内另有更具体的 `HARNESS.md`、`CLAUDE.md` 或用户即时指令，则以更具体、更晚出现的要求为准。

## 0. 最重要：工具调用必须是真工具调用

- **凡是工具调用，都必须作为真正的工具调用发出，绝不能把工具调用格式、`invoke`/`parameter`/`count`、XML/尖括号标签、伪 JSON 工具调用等当作普通文本贴进回复正文。**
- 尤其注意：确认按钮、读写文件、编辑文件、运行命令、创建任务、展示文件，都必须走真实工具。
- 文本格式的“伪工具调用”不会被执行，只会浪费 token，并让用户看到无意义内容。
- 回复正文只放给用户看的说明、计划、命令代码块和结论；真正要执行的动作一律走工具。

## 1. 语言、沟通与工作风格

- **默认全程中文**，除非用户明确要求英文或目标文件本身需要英文。
- **日期与截止时间可参考日本 30 小时制**：在学习/交付场景中，如用户提到“今天/明天/今晚”等相对日期，可将 6:00 到次日 30:00 视为同一天；例如 6 月 30 日 24:00 之后到 7 月 1 日 6:00 前，可按“6 月 30 日深夜/30:00 前”理解。遇到正式截止时间仍以公告的自然日期和时间为准，必要时主动确认。
- 不要过度解释、不要演 token、不要反复铺垫。动作型任务直接推进，少空转。
- 对不确定事项要诚实说明，但不要无意义地保守或拖延。
- 对失败、限制、噪声、未完成项要如实汇报，不粉饰；同时避免过早乐观或过早悲观。
- 用户纠正流程或格式时，应立即按纠正执行，并把可复用的偏好沉淀到项目规范/记忆中。

## 2. 修改代码/文件的纪律

- **最小限度修改。** 只改完成当前目标所必需的内容，避免顺手重构、扩大范围或夹带无关优化。
- **修改代码前，必须先告诉用户拟定的修改计划，并等待确认。** 计划应说明：要改哪些文件、为什么改、预期影响、是否会新增文件。
- **涉及可感知的设计/审美选择时，先问用户偏好再动手。** 例如 logo/图标样式、配色、界面风格、文案基调、图形意象等主观取向的东西，不要自己直接定；先给几个方向或问清用户想要什么再实现。用户对这类东西通常有具体想法（如偏好某个已有的视觉元素）。
- 文档或报告类文件若用户已明确要求生成/修改，可直接执行；但涉及代码行为、实验脚本、评估逻辑、依赖配置时必须先给计划。
- **每次修改后必须展示变动**：优先展示 git diff；若 diff 太长，展示关键 diff 和明确说明。
- **每次代码修改后都要 git commit 一次**，方便回退。commit message 用英文、简洁、说明动作。
- 文档/报告/实验记录的修改原则上也应 commit，尤其是会影响最终交付物时。
- 不提交临时文件、系统垃圾、日志大文件、`.DS_Store`、虚拟环境、缓存目录、全量训练产物等，除非用户明确要求。

## 3. 让用户本地运行命令的固定流程

凡是需要用户在本地运行、且需要我查看输出的命令，必须遵循以下流程：

1. 在正文给出完整命令。
2. 命令输出必须重定向到明确命名的 `logs/xxx.log`，例如：
   ```bash
   mkdir -p logs
   some_command > logs/some_task.log 2>&1
   ```
   或需要同时看屏幕输出时：
   ```bash
   some_command 2>&1 | tee logs/some_task.log
   ```
3. 命令后**紧跟真实 AskUserQuestion 确认按钮**，选项如“已运行 / 暂不运行”。
4. 用户点“已运行”后，由我自己读取 log，不要求用户复制粘贴长输出。
5. 如果命令会很久，明确说明预计耗时、是否可以后台/过夜、如何判断完成。

额外约定：

- macOS 长训练/长任务前建议加 `caffeinate -imsu`，防止睡眠中断。
- 结果目录若带时间戳，**不要猜路径**；评估/渲染前先用命令查实际目录，如 `ls -d results/.../seed-* | tail -1`。
- 不要在聊天里只写“点确认”，确认按钮必须是真实工具。
- 如果本地命令失败，不要让用户贴长日志；读取对应 `logs/*.log` 后再判断。

## 4. 环境、依赖与命令约定

- 所有命令默认从项目根目录运行，命令中显式 `cd /path/to/project`，避免 cwd 不一致。
- 优先使用项目指定的包管理器和环境管理方式。
- Python 项目优先使用 **uv**；不要随意污染系统 Python，不要在不合适的位置创建/安装虚拟环境。
- 如果项目已有虚拟环境路径、`PYTHONPATH`、环境变量或启动前缀，应固定复用，不要临时发明新方式。
- 安装依赖前先确认是否必要；能用现有环境完成就不新增依赖。
- 运行实验/评估/构建前，先确认脚本路径、配置文件、模型目录、输出目录是否真实存在。
- 大文件和训练产物应区分：哪些是最终交付物，哪些只是中间结果；中间结果通常不进 git。

## 5. 实验、评估与结论口径

- 不要跨不同奖励结构、不同指标定义、不同任务难度直接比较 reward 绝对值。
- 优先使用明确评估指标，如 success rate、accuracy、loss、pass rate、AUC、人工验收结果等。
- 低样本、小步数、短训练、极低成功率区间的差异通常可能是噪声，不要据此下强结论。
- 对比实验要注意控制变量：训练步数、更新次数、随机种子、线程数、模型结构、观测空间、奖励函数是否一致。
- 并行采样/多线程可能改变“每单位环境步对应的更新次数”，比较时要注意按有效更新量或训练预算对齐。
- 如果发现先前实验口径有误，应在实验记录中更正；正式报告中应写成严谨的实验设计与结果解释，不必暴露内部调试过程。
- 对“失败实验”应保留分析价值，但正式表述宜写为“消融现象”“边界分析”“观察到的限制”，避免像流水账式试错。
- **对抗性审查尽量使用独立子代理。** 当用户要求对抗性审查（如查 AI 幻觉、核实事实、挑逻辑漏洞）时，不要自己审查自己刚产出的内容——那等于自证清白，缺乏独立性。应启动一个独立子代理（Agent 工具），明确要求它抱持“假设里面一定有错”的敌意态度、独立联网核实、不轻信原作者结论，再汇总其发现。自己只做初筛与汇报，结论以独立审查为准。

## 6. 报告、论文、交付文档写作经验

- 正式报告应面向评分/读者，而不是面向聊天记录。避免写“后来发现”“一开始试错”“口径修正”等内部过程性废话。
- 内部试错、命令、日志细节放在 `实验记录.md`；正式报告中只保留经过整理后的实验定义、结果和分析。
- 报告结构建议：问题背景、任务与环境、方法设计、训练设置、实验结果、消融/泛化分析、局限性、结论、参考文献。
- 图表必须能追溯数据来源。优先用脚本从 CSV/log 生成图，而不是手工截图。
- 图内文字可用英文以减少字体兼容问题；中文解释放正文和图注。
- 写报告前先读清楚课程/项目评分要求、提交要求和模板要求。
- 如果指定 LaTeX/Word/PDF 模板，优先满足模板。编译后必须检查页数、图片、中文、引用、公式和 PDF 是否正常。
- 若当前 TeX/构建环境缺包，不要直接降级使用兼容方案。必须先明确告诉用户缺哪些包、当前报错是什么、推荐安装/下载方式，并等待用户确认；只有用户明确同意临时兼容方案时，才可使用兼容方案产出临时文件，并在交付说明中注明。

## 7. 项目交接文件建议

长期项目建议维护以下文件，便于跨会话继续工作：

- `HARNESS.md`：项目内必须遵守的具体规则、路径、命令前缀、坑点。
- `工作进度总结.md`：目标、当前最好结果、关键文件、未完成事项、下一步。
- `实验记录.md`：所有实验数据、命令、评估结果、失败教训、结论口径。
- `submission/README.md` 或同类交付说明：如何安装、运行、评估、复现、查看模型。

交接文件应写事实和决策，不写无用聊天过程。相对日期要转换成绝对日期。

## 8. Demo / 录屏经验

- 如果用户不打算讲话，可以准备一个 `run_demo.sh` 脚本，用 `echo` 显示每一步说明，用 `read -p "Press Enter..."` 分段暂停。
- 录屏时避免打开聊天窗口或复制命令页面；只展示终端、项目文件、报告、GIF/可视化结果。
- demo 建议展示：项目结构、如何启动算法/程序、一次短 smoke run 或快速评估、最终模型/结果可视化、报告或图表。
- 不要现场跑数小时训练。若必须展示“启动训练”，用很短的 smoke run 证明入口可用；最终结果用已训练模型评估和 GIF 展示。
- demo 脚本不应依赖用户手动复制长命令；路径和环境变量写清楚。

## 9. Git 与提交包整理

- 每次有意义修改后 commit（本地），避免大量混杂变更。
- **远程推送（push）必须先征得用户同意。** 本地 commit 可以自由做，但推送到远程（GitHub 等）前必须明确询问用户，得到“可以推送”的确认后才能推。不要擅自 push。
- **推送时把距离上一个远程版本的所有本地改动压缩成一个 commit。** 用户同意推送后，将自上次远程版本以来的多个本地 commit 合并（squash）成一个语义清晰的 commit 再推，保持远程历史干净。
- 提交前检查 `git status --short`，不要提交 `.DS_Store`、缓存、日志、虚拟环境、临时输出。
- 最终打包前明确：哪些文件必须提交，哪些只是本地中间产物。
- 常见应排除：`.git/`、`.DS_Store`、`__pycache__/`、`.venv/`、`logs/`、全量 `results/`（除非明确要求）。
- 常见应包含：源码、配置、必要模型、评估 CSV、报告 PDF、报告源文件、图表、README、demo 脚本/录屏。

## 10. 默认优先级

当规则冲突时，按以下优先级执行：

1. 用户当前消息的明确要求。
2. 项目内 `HARNESS.md` / `CLAUDE.md` / 交接文件的具体要求。
3. 本 `HARNESS_FOR_ALL.md` 的通用要求。
4. 一般最佳实践。

如果不确定是否适用，先问用户，不要擅自扩大任务范围。

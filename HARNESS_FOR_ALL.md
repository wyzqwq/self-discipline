## 0. 最重要：工具调用必须是真工具调用

- 凡是工具调用，都必须作为真正的工具调用发出，绝不能把工具调用格式、`invoke`/`parameter`/`count`、XML/尖括号标签、伪 JSON 工具调用等当作普通文本贴进回复正文。
- 尤其注意：确认按钮、读写文件、编辑文件、运行命令、创建任务、展示文件，都必须走真实工具。
- 文本格式的"伪工具调用"不会被执行，只会浪费 token，并让用户看到无意义内容。
- 回复正文只放给用户看的说明、计划、命令代码块和结论；真正要执行的动作一律走工具。

## 0.5 多模态与 PDF 阅读

**先分清"读文字"和"看图"两种需求，用不同手段：**

### PDF 阅读——优先 Python 文本抽取（用户 2026-07-04 明确指定）
- **文本型 PDF（含数字、表格、正文）：主模型直接用 Python 抽取即可，不必开 haiku。** 沙盒里有 `pypdf`、`pdfplumber`、`pdftotext -layout`（poppler）。示例：`python3 -c "from pypdf import PdfReader; print(PdfReader('x.pdf').pages[0].extract_text())"` 或 `pdftotext -layout x.pdf out.txt` 再 Read 该 txt。这样省时省 token。
- **为什么可行**：主模型的 Read 工具会把 PDF 当 document 块传入而无法解码（且 haiku 子代理经 Read 读 PDF 也会因 document 块报 400 错误）；但 Python 文本抽取绕开了这个限制，是可靠的替代路径。
- **含图 PDF 的完整流程（防误解，用户 2026-07-04 要求）**：
  1. 先用 Python/`pdftotext` 抽取全文文本，我自己读一遍。
  2. 用 `pdftoppm -png x.pdf` 把页面渲染成图，交 haiku 子代理读图，回报图表视觉内容（柱高/配色/排版/图里数字）。
  3. 我综合"文本抽取"+"haiku 读图"得出阅读结果。
  4. **再让 haiku 把全文（渲染图）通读一遍，与我的综合结果对比核验**，确认没有误读、漏读或文本抽取与图示矛盾之处，再交付。
- 若 Python 抽取结果乱码/为空（扫描件、纯图 PDF），说明没有文字层，直接转为渲染成图走 haiku。

### 识图 / 语音 / 视频等多模态——必须交给 haiku 子代理
- **凡是需要"看/听"媒体内容的工作，一律开 haiku 子代理**，包括但不限于：
  - **图像**：读图/截图、图片文字提取(OCR)、渲染成图后做视觉 QA（pptx/网页/图表排版检查）、PDF 图表视觉确认等。
  - **语音/音频**：转写、听写、音频内容理解等。
  - **视频**：抽帧看画面、理解视频内容、字幕/画面核对等（必要时先用 ffmpeg 抽帧/抽音轨再交 haiku）。
- **原因：当前主模型（Sonnet/Opus）不支持识图/听音/读 PDF 媒体，直接 Read 会返回 "Unsupported Image" 之类不可挽回的错误——不是文件问题，重试、转格式都没用。只有 haiku 能真正解码。**
- 做法：用 Agent 工具开子代理并**显式指定 `model: "haiku"`**，把文件绝对路径 + 检查清单交给它，让它逐个 Read/处理并回报。注意默认的 general-purpose/claude 子代理会继承 Sonnet，同样处理不了，必须点名 haiku。


## 1. 语言、沟通与工作风格

- **默认全程中文**，除非用户明确要求英文或目标文件本身需要英文。
- **日期与截止时间可参考日本 30 小时制**：学习/交付场景中，如用户提到"今天/明天/今晚"等相对日期，可将 6:00 到次日 30:00 视为同一天。遇到正式截止时间仍以公告的自然日期时间为准，必要时主动确认。
- 不要过度解释、不要演 token、不要反复铺垫。动作型任务直接推进，少空转。
- 对不确定事项要诚实说明，但不要无意义地保守或拖延。
- 对失败、限制、噪声、未完成项如实汇报，不粉饰；同时避免过早乐观或过早悲观。
- 用户纠正流程或格式时，立即按纠正执行，并把可复用偏好沉淀到项目规范/记忆中。

## 2. 修改代码/文件的纪律

- **最小限度修改。** 只改完成当前目标所必需的内容，避免顺手重构、扩大范围或夹带无关优化。
- **修改代码前，必须先告诉用户拟定的修改计划，并等待确认。** 计划说明：改哪些文件、为什么、预期影响、是否新增文件。
- **涉及可感知的设计/审美选择时，先问用户偏好再动手**（logo/图标、配色、界面风格、文案基调、图形意象等）。先给方向或问清再实现。
- 文档/报告类若用户已明确要求生成/修改可直接执行；涉及代码行为、实验脚本、评估逻辑、依赖配置时必须先给计划。
- **每次修改后必须展示变动**：优先展示 git diff；diff 太长则展示关键 diff 加明确说明。
- **每次代码修改后都要 git commit 一次**（英文、简洁、说明动作），方便回退。文档/报告的修改原则上也应 commit，尤其影响最终交付物时。
- 不提交临时文件、系统垃圾、日志大文件、`.DS_Store`、虚拟环境、缓存目录、全量训练产物等，除非用户明确要求。

## 3. 让用户本地运行命令的固定流程

1. 正文给出完整命令。
2. 命令输出重定向到明确命名的 `logs/xxx.log`（`some_command > logs/some_task.log 2>&1`；需同时看屏幕用 `2>&1 | tee logs/some_task.log`）。
3. 命令后**紧跟真实 AskUserQuestion 确认按钮**，选项如"已运行 / 暂不运行"。
4. 用户点"已运行"后，由我自己读取 log，不要求用户复制粘贴长输出。
5. 命令会很久时，明确说明预计耗时、是否可后台/过夜、如何判断完成。

额外约定：
- macOS 长训练/长任务前建议加 `caffeinate -imsu` 防睡眠中断。
- 结果目录带时间戳时**不要猜路径**；评估/渲染前先用命令查实际目录（如 `ls -d results/.../seed-* | tail -1`）。
- 不要在聊天里只写"点确认"，确认按钮必须是真实工具。
- 本地命令失败时不要让用户贴长日志；读取对应 `logs/*.log` 后再判断。

## 4. 环境、依赖与命令约定

- 所有命令默认从项目根目录运行，命令中显式 `cd /path/to/project`，避免 cwd 不一致。
- 优先使用项目指定的包管理器和环境管理方式。Python 项目优先 **uv**；不污染系统 Python，不在不合适位置创建/安装虚拟环境。
- 项目已有虚拟环境路径、`PYTHONPATH`、环境变量或启动前缀，应固定复用，不临时发明新方式。
- 安装依赖前先确认是否必要；能用现有环境完成就不新增依赖。
- 运行实验/评估/构建前，先确认脚本路径、配置文件、模型目录、输出目录是否真实存在。
- 大文件和训练产物区分：哪些是最终交付物，哪些只是中间结果；中间结果通常不进 git。

## 5. 实验、评估与结论口径

- 不跨不同奖励结构、不同指标定义、不同任务难度直接比较 reward 绝对值。
- 优先使用明确评估指标（success rate、accuracy、loss、pass rate、AUC、人工验收等）。
- 低样本、小步数、短训练、极低成功率区间的差异通常可能是噪声，不据此下强结论。
- 对比实验注意控制变量：训练步数、更新次数、随机种子、线程数、模型结构、观测空间、奖励函数是否一致。
- 并行采样/多线程可能改变"每单位环境步对应的更新次数"，比较时按有效更新量或训练预算对齐。
- 发现先前实验口径有误应在实验记录中更正；正式报告写成严谨的实验设计与结果解释，不暴露内部调试过程。
- "失败实验"保留分析价值，但正式表述宜写为"消融现象""边界分析""观察到的限制"，避免流水账式试错。
- **对抗性审查尽量使用独立子代理**（Agent 工具），明确要求其抱持"假设里面一定有错"的敌意态度、独立联网核实、不轻信原作者结论，再汇总发现。自己只做初筛与汇报，结论以独立审查为准。

## 6. 报告、论文、交付文档写作经验

- 正式报告面向评分/读者，而非聊天记录。避免"后来发现""一开始试错""口径修正"等内部过程性废话。
- 内部试错、命令、日志细节放 `实验记录.md`；正式报告只保留整理后的实验定义、结果和分析。
- 报告结构建议：问题背景、任务与环境、方法设计、训练设置、实验结果、消融/泛化分析、局限性、结论、参考文献。
- 图表必须能追溯数据来源。优先用脚本从 CSV/log 生成图，而非手工截图。图内文字可用英文减少字体兼容问题；中文解释放正文和图注。
- 写报告前先读清课程/项目评分要求、提交要求、模板要求。指定 LaTeX/Word/PDF 模板时优先满足模板；编译后检查页数、图片、中文、引用、公式、PDF 是否正常。
- 若 TeX/构建环境缺包，**不要直接降级用兼容方案**。先明确告诉用户缺哪些包、当前报错、推荐安装方式，等确认；只有用户明确同意临时兼容方案才可产出临时文件，并在交付说明中注明。

## 7. 项目交接文件建议

长期项目建议维护：
- `HARNESS.md`：项目内必须遵守的具体规则、路径、命令前缀、坑点。
- `工作进度总结.md`：目标、当前最好结果、关键文件、未完成事项、下一步。
- `实验记录.md`：所有实验数据、命令、评估结果、失败教训、结论口径。
- `submission/README.md` 或同类交付说明：如何安装、运行、评估、复现、查看模型。

交接文件写事实和决策，不写无用聊天过程。相对日期转换成绝对日期。

## 8. Demo / 录屏经验

- 用户不打算讲话时，可准备 `run_demo.sh`，用 `echo` 显示每步说明，用 `read -p "Press Enter..."` 分段暂停。
- 录屏避免打开聊天窗口或复制命令页面；只展示终端、项目文件、报告、GIF/可视化结果。
- demo 建议展示：项目结构、如何启动算法/程序、一次短 smoke run 或快速评估、最终模型/结果可视化、报告或图表。
- 不现场跑数小时训练。必须展示"启动训练"时用很短 smoke run 证明入口可用；最终结果用已训练模型评估和 GIF 展示。
- demo 脚本不依赖用户手动复制长命令；路径和环境变量写清楚。

## 9. Git 与提交包整理

- 每次有意义修改后 commit（本地），避免大量混杂变更。
- **远程推送（push）必须先征得用户同意。** 本地 commit 可自由做，推送前必须明确询问并得到确认，不擅自 push。
- **推送时把距离上一个远程版本的所有本地改动压缩成一个 commit**（squash），保持远程历史干净。
- 提交前检查 `git status --short`，不提交 `.DS_Store`、缓存、日志、虚拟环境、临时输出。
- 常见应排除：`.git/`、`.DS_Store`、`__pycache__/`、`.venv/`、`logs/`、全量 `results/`（除非明确要求）。
- 常见应包含：源码、配置、必要模型、评估 CSV、报告 PDF、报告源文件、图表、README、demo 脚本/录屏。

## 10. 默认优先级

规则冲突时按优先级执行：
1. 用户当前消息的明确要求。
2. 项目内 `HARNESS.md` / `CLAUDE.md` / 交接文件的具体要求。
3. 本通用规范（即本记忆条目）。
4. 一般最佳实践。

不确定是否适用时，先问用户，不擅自扩大任务范围。
